### Step 5.3 — Escalation Engine (commit `99677d5`)

**New files:**
- `src/app/api/escalation/cron/route.ts` — Automated escalation runner
- `src/app/api/escalation/log/route.ts` — GET (list) + PATCH (resolve)
- `src/app/admin/escalations/page.tsx` — Admin UI for escalation log

**Modified files:**
- `src/lib/db-v5.ts` — Added `markReadinessItemEscalated()`, `listEscalations()`
- `src/types/index.ts` — Updated `EscalationLogEntry` with full DB fields

**Escalation chain (4 levels):**
1. **Level 1**: Post warning in patient thread channel ("⚠️ Overdue: [item] for [patient]")
2. **Level 2**: Escalate to department head (lookup via `getDepartmentHead()`)
3. **Level 3**: Escalate to on-duty staff (via `/api/duty-roster/resolve` — this is why 5.2 had to be built first)
4. **Level 4+**: Broadcast to `emergency-escalation` cross-functional channel

**Key mechanics:**
- 60-minute cooldown between re-escalation of the same item (`escalated_at` timestamp check)
- Auth: `CRON_SECRET` header (for Vercel cron) OR super_admin session
- Cron finds overdue readiness items: `status = 'pending'` AND `sla_deadline < now()`
- Each escalation creates an `escalation_log` entry and increments `escalation_level` on the readiness item
- Admin page has "Run Escalation Check" button for manual trigger
- Resolve modal: mark escalation resolved with notes

**Known edge case found during testing:** Cron found 1 overdue item with a patient-thread channel (`pt-b4e6713b`) that didn't exist in GetStream — this was leftover test data from earlier sessions. The engine caught the error gracefully and logged it without crashing.

### Step 6.2 — UX Redesign: Bottom Tab Bar Navigation (commit `f9044b1`)

**Background:** V tested the live app after Steps 5.1-6.1 were deployed and found it unintuitive. Key problems:
- System messages in chat weren't actionable (just text, no buttons)
- No UI to create patient threads (had to know the API)
- Forms weren't discoverable from chat context
- No onboarding — blank screen after first login
- Chat-first layout didn't match hospital staff mental model (they think in patients, not channels)

**Decision:** Redesign UX before alpha release. Move from chat-first to patient-first navigation.

**New files (6):**
- `src/components/AppShell.tsx` (69 lines) — New main wrapper, replaces ChatPage as entry point
- `src/components/layout/BottomTabBar.tsx` (57 lines) — Fixed bottom nav with 4 tabs

- `src/components/tasks/TasksView.tsx` (228 lines) — Overdue items + escalations
- `src/components/profile/ProfileView.tsx` (137 lines) — Profile card + admin link + logout

**Modified files (2):**

- `src/components/chat/ChatShell.tsx` — Changed 3x `h-screen` → `h-full` (required for tab layout; h-screen would overflow past the bottom tab bar)

**Architecture: AppShell pattern**
```
AppShell (wraps ChatProvider)
├── Tab Content Area (flex-1 overflow-hidden)
│   ├── PatientsView    (shown when activeTab === 'patients')
│   ├── ChatShell       (shown when activeTab === 'chat') ← ALWAYS MOUNTED via `hidden` class
│   ├── TasksView       (shown when activeTab === 'tasks')
│   └── ProfileView     (shown when activeTab === 'me')
└── BottomTabBar (fixed bottom)
```

**Critical design choice:** ChatShell is ALWAYS mounted (uses CSS `hidden` class to toggle visibility, NOT conditional rendering). This keeps the GetStream WebSocket connection alive when user switches to other tabs. If ChatShell unmounted/remounted, it would reconnect every time — slow and expensive.

**Tab bar details:**

- Active indicator: blue line at top of active tab, blue icon + text
- Badge support on tabs (for unread counts — wiring pending)
- `safe-area-bottom` padding for iPhone notch/home indicator

**PatientsView details:**

## Architectural Decisions Made in This Thread

1. **Patient-first, not chat-first**: The original design put chat at the center. After live testing, V confirmed hospital staff think in terms of patients, not channels. Patients tab is now the default landing screen.

2. **WhatsApp-like tab pattern**: Chose bottom tab bar (not sidebar, not top tabs) because hospital staff are most familiar with WhatsApp. Four tabs keeps it simple.

3. **AppShell replaces ChatPage**: Rather than modifying ChatPage to add tabs, created a new AppShell wrapper that contains ChatShell as one of four tab views. ChatPage still exists but is orphaned — it's the old entry point.

4. **CSS hidden toggle, not conditional render**: For tab switching, we use `className={activeTab === 'chat' ? 'h-full' : 'hidden'}` instead of `{activeTab === 'chat' && <ChatShell />}`. This is critical because GetStream's WebSocket connection lives inside ChatShell and we don't want to reconnect every time the user switches tabs.

5. **Escalation before UX**: Built the full escalation engine (5.3) before the UX redesign (6.2) so that the Tasks tab had real data to display from day one.

6. **Duty roster feeds escalation**: The on-duty resolution endpoint from the duty roster (5.2) is a direct dependency of escalation level 3 (5.3). Build order was intentional.
